# PRD：AI 运营数据分析智能体平台

> 基线骨架：[Coder42Y/full-stack-ai-agent-template](https://github.com/Coder42Y/full-stack-ai-agent-template)
> 目标岗位：哈啰普惠 AI 全栈开发工程师（校招） 22-30K 上海
> 编写日期：2026-06-07

---

## 1. 问题陈述

### 场景：一个调度员的早晨

> 早上 6:30，共享出行公司的调度员老李打开手机。昨天暴雨，他心里没底：哪些区域的单车被淹了需要清运？今天早高峰张江地铁站够不够车？徐家汇商圈昨晚是不是又堆了上百辆没人骑？
>
> 他想查数据，但能用的工具只有 BI 后台。他不会写 SQL，只能在十几个固定报表里来回翻——翻到了也不一定能回答他的问题，因为这些报表是上周的数据，不是实时状态。
>
> 最后，老李凭经验和直觉排了今天的调度计划。9 点，张江地铁站果然爆单——他低估了客流。运营经理在工作群 @ 他：「张江没车了，用户投诉。」

### 真正的痛点不是"不会写 SQL"

是运营决策和实时数据之间的**距离**。这个距离包括：

| 距离 | 具体表现 | 后果 |
|------|---------|------|
| **工具距离** | 数据在数据库里，调度员够不着 | 凭经验拍脑袋，早上猜错了全天乱 |
| **时间距离** | 从「想知道」到「知道答案」等半天 | 暴雨来了不知道哪要紧急清运，等数据出来雨都停了 |
| **解读距离** | 看到数字，但不知道意味着什么 | 看到某区域车辆数 200，不知道这是正常还是堆积 |
| **行动距离** | 知道问题了，不知道该怎么处理 | 发现异常 → 再翻运营手册 → 再问同事 → 半小时过去了 |

### 为什么现有方案不行

**BI 工具（Tableau、Metabase）**：你首先得知道你想看什么、知道怎么拖字段。它擅长回答「已知问题」——"这个月上海订单量多少"；但回答不了「探索式问题」——"我觉得最近哪里不对劲，帮我看看怎么回事"。

**数据团队排期**：一个「帮我查个数」的请求，排上队就是半天。调度员的决策窗口只有几十分钟。

**自建 AI 查询界面**：各公司都在做，但每个都是和自家数据库紧耦合的定制方案。换一家公司、换一套表结构，代码要重写。

### 机会

LLM + MCP 可插拔工具调用，天然适合解决"运营决策数据和人的最后一公里"：
- **LLM** 理解自然语言意图、生成分析推理
- **MCP** 让数据源变成可插拔的标准件——换公司换行业只换配置不换架构
- **Agent 编排**（路由、并行、条件分支）处理运营场景的多步骤分析链路

## 2. 解决方案

### 一句话

老李早上打开的不是 BI 后台，是一个对话框。他问：「昨晚暴雨，帮我看下哪些区域车辆需要清运，顺便告诉我张江地铁站今天早高峰够不够车。」Agent 自动查了车辆分布表和天气表，标记了 3 个积水高风险区域需要清运，又对比了张江最近两周的早高峰用车数据，返回：**「够，但建议比平时多备 20 辆，因为今天是周一 + 昨晚下雨地面湿滑，单车需求会比电单车高。」**

### Demo 行业：共享出行运营

选择共享出行（共享单车/助力车）作为 Demo 行业，原因：
1. 直接匹配目标公司（哈啰）的业务领域，面试官 10 秒建立共鸣
2. 数据场景丰富：订单、车辆分布、区域调度、天气影响、定价
3. 运营角色明确：区域经理、调度员、数据分析师
4. 决策链路长但可模拟：从「发现问题」到「分析数据」到「生成行动建议」全流程在 Agent 内闭环

### 平台架构（行业无关）

```
用户自然语言提问
       │
       ▼
┌──────────────────────────────┐
│  Supervisor Agent            │  ← 识别意图：查数据 / 看趋势 / 异常检测 / 知识问答
│  (LangGraph StateGraph)      │
└──────────┬───────────────────┘
           │ 路由
     ┌─────┼─────┬──────────┐
     ▼     ▼     ▼          ▼
  查数据  看趋势  异常检测   知识问答
     │     │     │          │
     └─────┴─────┴──────────┘
           │ MCP 协议调用
     ┌─────┼──────────┐
     ▼     ▼          ▼
  PostgreSQL  Tavily   图表生成
  (NL2SQL)  (Web搜索)  (ECharts)
     │
     ▼
  ┌──────────────────────┐
  │  输出：              │
  │  · 数据表格          │
  │  · 趋势图表          │
  │  · 异常标记          │
  │  · 分析文字          │
  │  · 运营建议          │
  └──────────────────────┘
```

### 消除四个距离

| 距离 | 现状（老李的早晨） | 用了 Agent 之后 |
|------|-------------------|----------------|
| **工具距离** | 数据在 PostgreSQL 里，够不着 | 用自然语言对话，零工具学习成本 |
| **时间距离** | 想知道 → 知道答案，等半天 | 实时查询，30 秒内返回结果 |
| **解读距离** | 看到数字，不知道意味着什么 | Agent 自动对比历史基线、标记异常、写分析结论 |
| **行动距离** | 知道问题，不知道怎么处理 | Agent 检索运营知识库，输出具体操作步骤 |

## 3. 用户故事

### Demo 场景用户

**角色 A：区域运营经理（小王）**

1. 作为区域运营经理，我想问"上个月浦东新区的日均订单量是多少"，Agent 直接告诉我数字，不用我打开后台翻报表
2. 作为区域运营经理，我想问"最近一周早晚高峰的用车趋势"，Agent 给我一张趋势折线图，而不是扔给我一个 CSV
3. 作为区域运营经理，我想问"哪些区域最近 24 小时有车辆堆积（超过 50 辆未移动）"，Agent 自动标记异常区域并给出清运建议
4. 作为区域运营经理，我想问"明天早 8 点张江地铁站大概需要多少辆车"，Agent 综合历史数据 + 天气 + 工作日类型给出预估
5. 作为区域运营经理，我想问"暴雨天气下调度策略要注意什么"，Agent 从运营知识库中检索并生成 checklist

**角色 B：调度员（小张）**

6. 作为调度员，我想问"现在徐家汇商圈车辆缺口最大的 3 个投放点"，Agent 实时查数据库返回排序列表
7. 作为调度员，我想问"今天下午的调度任务优先级怎么排"，Agent 综合分析车辆分布 + 历史需求 + 天气给出建议
8. 作为调度员，我想收到异常告警推送——当某个区域车辆堆积超过阈值时自动通知我

**角色 C：数据分析师（小李）**

9. 作为数据分析师，我想问"对比本月和上月各区域的订单增长率"，Agent 返回对比表格 + 增长率热力图
10. 作为数据分析师，我想问"哪种车型的用户复购率最高"，Agent 自动写 SQL 查询并返回结果
11. 作为数据分析师，我想问"下雨天 vs 晴天的单车使用量差异"，Agent 关联天气表和订单表做交叉分析

### 平台能力用户

**角色 D：Prompt 模板管理员**

12. 作为管理员，我想创建不同的 Prompt 模板（巡检模式/分析模式/应急模式），让 Agent 在不同场景下给出不同风格的回答
13. 作为管理员，我想一键激活某个 Prompt 模板，所有用户立刻生效，不用重新部署
14. 作为管理员，我想在创建模板时定义变量占位符（如 `{{region}}`、`{{date}}`），模板激活时自动填充

**角色 E：平台运维**

15. 作为运维人员，我想在 `.env` 中配置 MCP Server 的地址和参数，Agent 启动时自动连接
16. 作为运维人员，我想禁用某个 MCP Server 时 Agent 自动回退到内置能力，不影响核心功能
17. 作为运维人员，我想看 Agent 的工具调用日志（哪个 MCP Server 被调了几次、耗时多少）

## 4. 实现决策

### 4.1 技术栈

| 层 | 选择 | 决策理由 |
|----|------|---------|
| 全栈骨架 | `full-stack-ai-agent-template`（已 fork） | 提供 FastAPI + Next.js + RAG + Celery 全套基础设施 |
| Agent 框架 | LangGraph | StateGraph + 条件路由天然适合 Supervisor Agent 拓扑 |
| LLM | PydanticAI + OpenAI API | 骨架自带的 Agent 封装；面试时可切换 DeepSeek 展示跨模型能力 |
| NL2SQL 引擎 | Agent 的 MCP Tool（LLM 生成 SQL → 执行） | 不需要额外 NL2SQL 引擎，LLM 写 SQL 的准确率对 Demo 数据足够 |
| MCP 协议 | Python `mcp` SDK（stdio transport） | 标准化的工具调用协议，JD 明确加分项 |
| 数据库 | PostgreSQL（有种子数据表） | 骨架自带，存 Demo 业务的订单/车辆/天气表 |
| 向量库 | Milvus | 骨架自带，存运营手册、调度策略等 RAG 文档 |
| 图表生成 | MCP Chart Server（ECharts） | 前端用 ECharts 渲染，MCP Server 返回 ECharts option JSON |
| 部署 | Docker Compose | 骨架自带 9 个服务的编排 |

### 4.2 为什么不从零写 NL2SQL 引擎

面试官可能会问。回答：

> "我的目标是验证 Agent 编排 + MCP 可插拔数据源的架构可行性，不是攻克 NL2SQL 的 SOTA 指标。LLM 写 SQL 对 Demo 规模的 schema（3-5 张表，每表 5-10 列）准确率很高。如果真的落地，我会接一个成熟的 NL2SQL 引擎（如 DataAgent 或开源方案），通过 MCP 协议替换掉现在这个 LLM-SQL tool，Agent 层的编排逻辑完全不需要改——这就是 MCP 的价值。"

### 4.3 Demo 数据设计

**不依赖外部真实数据**。全部用 LLM + Python 脚本生成。

| 表 | 行数 | 字段 | 生成逻辑 |
|----|------|------|---------|
| `orders` | 1000 条 | id, created_at, region, vehicle_type, amount, user_id | Python 脚本：随机区域(50个)+ 车型(3种)+ 金额(1-50元)+ 时间戳分布(早晚高峰权重大) |
| `vehicle_distribution` | 50 区域 × 24 小时 = 1200 条 | region, hour, bike_count, ebike_count | Python 脚本：模拟单车/电单车在 50 个区域的 24 小时分布曲线 |
| `weather` | 50 区域 × 7 天 = 350 条 | region, date, weather_type, temperature, precipitation | Python 脚本：随机天气(晴/阴/雨/暴雨)+ 温度(5-35°C) |

**生成命令**：

```bash
uv run ai_agent_test cmd seed-data
```

### 4.4 Agent 能力矩阵

| Agent 功能 | 实现方式 | 涉及 MCP Server |
|-----------|---------|----------------|
| 数据查询 | LLM 将自然语言翻译为 SQL → 执行 → 格式化返回 | `pg_query` Server |
| 趋势分析 | 同上 + 自动按小时/天聚合 + 生成图表 | `pg_query` + `chart` Server |
| 异常检测 | SQL 查异常阈值条件（如 vehicle_count > 50 AND last_update > 24h） | `pg_query` Server |
| 调度建议 | 先查历史数据 + 天气 + 工作日 → LLM 综合生成建议 | `pg_query` + `web_search`（查天气） |
| 知识问答 | RAG 检索运营手册 → LLM 生成答案 | 无 MCP（走骨架自带 RAG） |

### 4.5 Prompt 模板设计

**巡检模式（日常用）**：

```
你是共享出行运营助手。当前是 {{time}}，区域 {{region}}。
请用简洁的 bullet 形式汇报：
- 当前车辆分布概况
- 异常项（标红）
- 建议操作（按优先级排序）
```

**分析模式（做报表用）**：

```
你是数据分析助手。用户关注 {{metric}} 在 {{time_range}} 的趋势。
请提供：
- 数据摘要（表格）
- 趋势图表
- 同比/环比变化
- 趋势解读与洞察
```

**应急模式（暴雨/故障）**：

```
你处于应急响应模式。当前天气 {{weather}}。
请按 checklist 格式输出：
- 需要立即处理的区域（按紧急程度排序）
- 每个区域的具体操作步骤
- 需通知的人员角色
```

## 5. 测试决策

### 测试原则

- **只测外部行为，不测实现细节**：Agent 返回的结果对不对，而不是内部调了几次 LLM
- **Mock 慢的外部依赖**：测试时用 SQLite 替代 PostgreSQL、用 mock Tavily client 替代真实 API
- **核心路径必须覆盖**：NL2SQL 生成 → 执行 → 结果 → 用户能看到

### 测试范围

| 模块 | 测什么 | 怎么测 |
|------|--------|--------|
| `pg_query_server` | NL2SQL 生成是否语法正确、查询结果是否符合预期 | pytest + SQLite temp db |
| `chart_server` | 生成的 ECharts option 是否合法 JSON | pytest + JSON Schema 验证 |
| Supervisor Agent | 不同意图是否正确路由到对应 MCP Server | pytest + mock MCP Client |
| Prompt 模板 | CRUD + 激活 + 变量替换 + fallback | 已有 `test_prompt_templates.py` |
| MCP 集成 | Client 连接 / list_tools / call_tool / disconnect | 已有 `test_mcp_integration.py` |

### 不做测试的

- **LLM 输出的语义质量**：文本生成类不做自动化测试，人工抽检
- **前端 E2E**：如果时间不够，只做手动 smoke test + 录屏
- **性能/压力测试**：面向面试的 Demo 不需要

## 6. 范围外

以下内容**不在本 PRD 范围**，面试时可作为「如果继续做」的扩展点：

1. **真实多租户计费系统**：骨架已有 teams/billing，但 Demo 不需要激活
2. **接入真实哈啰数据库**：Demo 用 mock 数据，面试时说明「MCP 换数据源即可接入」
3. **移动端适配**：JD 不要求，骨架本身也没有移动端
4. **训练/微调自定义模型**：用现成 OpenAI API，不碰训练
5. **多语言支持**：只做中文
6. **多 Agent 协作调度**（超过 Supervisor 模式）：当前 Supervisor 是 1 对 N 路由，不做 Agent-to-Agent 直接通信
7. **实时数据推送/WebSocket 告警**：用户故事中有提，但 P2，有时间再说
