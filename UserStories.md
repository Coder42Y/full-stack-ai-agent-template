## E1：自然语言数据查询

> 核心价值：让不会 SQL 的运营人员用自然语言直接查到数据，消除"工具距离"。

---

### US-01：按区域查询聚合指标

**Card**

> 作为**区域运营经理**，我想用自然语言问"上个月浦东新区的日均订单量是多少"，Agent 直接返回一个具体数字，这样我不用打开后台翻报表也能知道结果。

**Conversation**

运营经理每天早上需要快速了解各区域的关键指标（订单量、车辆数、营收等）。目前这些数据散落在 BI 后台的多个报表中，需要切换多个页面、筛选条件才能拼出一个数字。Agent 应该能理解"上个月"、"浦东新区"、"日均订单量"这些自然语言表述，自动转换为 SQL 查询，直接给出答案。

**Acceptance Criteria**

1. 用户在对话框输入"上个月浦东新区的日均订单量是多少"后，Agent 在 30 秒内返回一个明确的数字回答（如"上个月浦东新区日均订单量为 1,234 单"）。
2. Agent 返回的结果附带原始 SQL 查询语句，供用户确认查询逻辑（可折叠展开）。
3. 当用户提到的区域名称在数据库中不存在时，Agent 返回模糊匹配建议列表（如"您说的是'浦东新区'吗？目前系统中有以下相似区域：…"）。
4. 当查询结果为空（该区域无数据）时，Agent 明确告知"未找到匹配数据"并建议用户检查区域名称或时间范围。
5. 查询结果展示区域名称、时间范围、指标名称和具体数值，结构清晰。

### US-02：实时查询车辆分布与缺口

**Card**

> 作为**调度员**，我想问"现在徐家汇商圈车辆缺口最大的 3 个投放点"，Agent 实时查数据库并返回排序列表，这样我能快速决定往哪里调车。

**Conversation**

调度员的核心工作是决定"往哪放车、从哪挪车"。他需要的是实时的、排序后的结果——不是原始数据。Agent 应该理解"车辆缺口最大"意味着"需求量 - 当前车辆数"的差值最大，并按缺口从大到小排序返回。

**Acceptance Criteria**

1. 用户输入查询后，Agent 返回 Top 3 投放点列表，每个投放点包含：名称、当前车辆数、预估需求量、缺口数。
2. 列表按缺口数从大到小排序。
3. 数据来自实时数据库查询（非缓存），查询时间戳在结果中展示。
4. Agent 在列表下方附带一句话运营建议（如"建议优先向徐家汇地铁 1 号口调度 15 辆车"）。
5. 当所有投放点车辆充足（无缺口）时，Agent 返回"当前所有投放点车辆充足，暂无调度需求"。

### US-03：跨表关联查询

**Card**

> 作为**数据分析师**，我想问"下雨天 vs 晴天的单车使用量差异"，Agent 自动关联天气表和订单表做交叉分析，这样我不用自己写 JOIN 语句。

**Conversation**

分析师经常需要跨表做关联分析（天气×订单、区域×车型等），这类查询 SQL 复杂、容易出错。Agent 应该能识别需要关联哪些表，自动生成 JOIN 查询，并以对比形式展示结果。

**Acceptance Criteria**

1. Agent 自动识别查询涉及 `weather` 表和 `orders` 表，生成带 JOIN 的 SQL。
2. 结果以对比表格形式展示：晴天订单量 vs 雨天订单量，分区域分行列出。
3. 表格底部附有汇总统计：晴天日均订单、雨天日均订单、差异百分比。
4. Agent 用文字解读数据含义（如"雨天订单量较晴天下降约 35%，主要受通勤时段影响"）。
5. 当关联条件不明确时（如用户未指定时间范围），Agent 追问用户确认而非猜测。

### US-04：按条件查询车辆堆积

**Card**

> 作为**区域运营经理**，我想问"哪些区域最近 24 小时有车辆堆积（超过 50 辆未移动）"，Agent 自动标记异常区域并给出清运建议，这样我能快速处理运营异常。

**Conversation**

车辆堆积（某区域大量车辆长时间未被使用）是共享出行运营的高频问题。运营经理需要的是"哪出了问题"+"该怎么做"一揽子答案，而不是查完数据再去翻运营手册。

**Acceptance Criteria**

1. Agent 查询 `vehicle_distribution` 表，筛选条件为：车辆数 > 50 且最近更新时间 > 24 小时。
2. 返回结果以列表形式展示，每个异常区域包含：区域名称、堆积车辆数、未移动时长。
3. 列表按堆积严重程度（车辆数 × 未移动时长）降序排列。
4. 每个异常区域附带一条清运建议（如"建议调配 2 辆清运车前往 XX 路口，预计 2 小时可完成"）。
5. 当无异常区域时，Agent 返回"当前无车辆堆积异常，一切正常"。

### US-05：多维度对比分析

**Card**

> 作为**数据分析师**，我想问"对比本月和上月各区域的订单增长率"，Agent 返回对比表格 + 增长率热力图，这样我能快速完成月度分析报告。

**Conversation**

分析师做月度/季度报告时，最常做的就是"本期 vs 上期"的对比分析。Agent 不只要返回数字，还要生成可视化图表，让分析师直接截图放进报告里。

**Acceptance Criteria**

1. Agent 自动计算每个区域的本月订单量、上月订单量、环比增长率（%）。
2. 结果以表格形式展示，按增长率从高到低排序。
3. Agent 同时生成 ECharts 热力图，颜色深浅表示增长率高低。
4. 表格和图表下方附有一段趋势解读文字（如"浦东新区增长率最高达 23%，主要受新投放点带动"）。
5. 增长率超过 ±50% 的区域被标记为"显著变化"，并附带可能原因分析。

### US-06：早晚高峰用车趋势图

**Card**

> 作为**区域运营经理**，我想问"最近一周早晚高峰的用车趋势"，Agent 给我一张趋势折线图和文字解读，这样我不用自己从 CSV 里画图。

**Conversation**

运营经理做周会汇报时，最常用的是趋势图——看看这周用车量是在涨还是在跌，和上周比怎么样。他需要一张"能直接贴到 PPT 里"的图，而不是一个 Excel 导出文件。

**Acceptance Criteria**

1. Agent 自动按"早高峰（7-9点）"和"晚高峰（17-19点）"分组聚合数据。
2. 生成 ECharts 折线图，X 轴为日期（最近 7 天），Y 轴为订单量，双折线分别展示早高峰和晚高峰。
3. 图表下方附文字解读，包含：整体趋势（上升/下降/持平）、最大值和最小值日期、与上周对比。
4. 图表支持鼠标悬停查看具体数值。
5. 当数据不足以绘制趋势（如某天数据缺失）时，Agent 标注缺失日期并说明影响。

### US-07：车型复购率分析

**Card**

> 作为**数据分析师**，我想问"哪种车型的用户复购率最高"，Agent 自动写 SQL 查询并返回结果表格和饼图，这样我能为产品策略提供数据支撑。

**Conversation**

产品团队需要了解不同车型（单车、助力车、电单车）的用户黏性。复购率的计算逻辑（同一用户在 N 天内再次下单）需要跨 `orders` 表聚合。Agent 应该能理解"复购率"的业务含义并生成正确的 SQL。

**Acceptance Criteria**

1. Agent 将"复购率"理解为"7 天内再次下单的用户占比"，生成对应的 SQL。
2. 结果以表格展示：车型、总用户数、复购用户数、复购率（%）。
3. 同时生成 ECharts 饼图，展示三种车型的复购率占比。
4. Agent 附带解读文字，指出复购率最高的车型并给出可能原因。
5. 用户可以追问"如果把复购窗口改为 30 天呢"，Agent 修改 SQL 重新查询。

### US-08：按区域×时段的用车热力图

**Card**

> 作为**数据分析师**，我想查看"各区域各时段的订单分布热力图"，Agent 生成可交互的热力图，这样我能一目了然地看出哪些区域在什么时段用车最多。

**Conversation**

热力图是运营分析中最直观的可视化方式——区域×时段的二维分布一眼就能看出"哪块热"。分析师需要这个图来做投放策略优化（在热点区域高峰前预投放车辆）。

**Acceptance Criteria**

1. Agent 查询 `orders` 表，按区域（行）和时段（列，0-23 小时）聚合订单量。
2. 生成 ECharts 热力图，颜色深浅表示订单量高低。
3. 热力图支持鼠标悬停查看具体数值（区域名 + 时段 + 订单量）。
4. Agent 在图表下方标注"Top 3 热点"（订单量最高的 3 个区域-时段组合）。
5. 区域数量超过 20 个时，热力图自动分页或提供滚动，避免图表过密不可读。

### US-09：供需预测

**Card**

> 作为**区域运营经理**，我想问"明天早 8 点张江地铁站大概需要多少辆车"，Agent 综合历史数据 + 天气 + 工作日类型给出预估值，这样我能提前准备调度。

**Conversation**

运营最怕的不是"出问题"，而是"没准备好"。提前知道明天某个区域需要多少车，就能提前调度。Agent 应该综合多种因素（历史同期数据、天气预报、工作日/节假日）给出一个有依据的预估值。

**Acceptance Criteria**

1. Agent 查询历史同时段数据（最近 4 周的早 8 点数据），取平均值作为基线。
2. 叠加天气因素（下雨天需求变化）和工作日类型（周一 vs 周五）的修正系数。
3. 返回预估值（如"预计需要 120 辆"）及置信度说明（如"基于最近 4 周历史数据，置信度中等"）。
4. 附带建议（如"建议比平时多备 20 辆，因为今天是周一 + 昨晚下雨地面湿滑，单车需求会比电单车高"）。
5. 当历史数据不足（新投放区域）时，Agent 明确标注"数据不足，预估值仅供参考"。

### US-10：综合调度优先级推荐

**Card**

> 作为**调度员**，我想问"今天下午的调度任务优先级怎么排"，Agent 综合分析车辆分布 + 历史需求 + 天气给出排序列表，这样我不用靠经验拍脑袋。

**Conversation**

调度员每天的核心决策就是"先调哪、后调哪"。现在他靠经验，但经验会出错（暴雨天和晴天的调度策略完全不同）。Agent 应该综合多个数据维度，给出一个有依据的优先级列表。

**Acceptance Criteria**

1. Agent 查询当前车辆分布、未来 3 小时历史需求曲线、天气预报三个数据源。
2. 返回调度任务列表，每条包含：目标区域、调度类型（投放/清运）、优先级（高/中/低）、建议车辆数、原因说明。
3. 列表按优先级排序，高优先级任务排在最前。
4. 恶劣天气（暴雨/大雪）自动提升相关区域优先级，并在原因中标注天气因素。
5. 调度员可以追问"如果把 XX 区域的优先级调高呢"，Agent 重新计算并调整排序。

### US-11：异常告警通知

**Card**

> 作为**调度员**，我想在某个区域车辆堆积超过阈值时收到告警通知，这样我不用一直盯着后台也能第一时间发现问题。

**Conversation**

调度员不可能 24 小时盯屏幕。他需要的是"有事叫我"——当出现车辆堆积、订单异常等状况时，系统主动通知他。这是从"人找数据"到"数据找人"的转变。

**Acceptance Criteria**

1. 系统每 15 分钟扫描一次 `vehicle_distribution` 表，检测车辆堆积（超过 50 辆且 24 小时未移动）。
2. 检测到异常时，在聊天界面推送一条告警消息，包含：异常区域、堆积数量、检测时间。
3. 告警消息附带快捷操作按钮："查看详情"（跳转到该区域的详细数据）和"标记已处理"。
4. 同一区域在 2 小时内不重复推送告警（去重）。
5. 告警阈值（车辆数、未移动时长、扫描频率）可在后台配置。

> **注：** 此故事为 P2 优先级，依赖 WebSocket 或前端轮询实现，可在基础版本后迭代。

### US-12：运营知识检索

**Card**

> 作为**区域运营经理**，我想问"暴雨天气下调度策略要注意什么"，Agent 从运营知识库中检索并生成 checklist，这样我不用翻运营手册也能知道标准流程。

**Conversation**

每个运营团队都有厚厚的运营手册，但真正遇到紧急情况时没人会去翻。Agent 把这些知识变成可对话、可检索的"活文档"。用户问一个问题，Agent 检索相关文档片段，生成结构化的回答。

**Acceptance Criteria**

1. Agent 通过 RAG 检索运营知识库（Milvus 向量库），返回与用户问题最相关的文档片段。
2. 回答以结构化格式呈现（checklist / 要点列表），不是大段文字。
3. 回答中标注引用来源（如"根据《恶劣天气运营预案 v2.1》第 3 节"）。
4. 当知识库中没有相关内容时，Agent 明确告知"知识库中暂无相关内容"，而非编造回答。
5. 用户可以追问"展开第 2 条"，Agent 检索更详细的子内容。

---

### US-13：调度策略 Checklist

**Card**

> 作为**调度员**，我想问"今天暴雨，我需要做哪些准备"，Agent 结合实时数据和运营知识库生成一份个性化的应急 checklist，这样我不遗漏任何步骤。

**Conversation**

应急场景下，调度员最怕的不是不知道怎么做，而是"漏了某一步"。Agent 应该结合当前的实际情况（哪些区域有暴雨、哪些区域车辆堆积），生成一份"今天这份"的个性化 checklist，而不是泛泛的标准流程。

**Acceptance Criteria**

1. Agent 同时查询实时数据（天气、车辆分布）和知识库（应急流程文档）。
2. 生成个性化 checklist，每条包含：待办事项、对应区域（如有）、紧急程度。
3. checklist 按紧急程度排序（立即处理 → 提前准备 → 持续关注）。
4. 涉及到具体区域的，标注该区域的实时数据（如"张江地铁站当前堆积 67 辆车"）。
5. 用户可以逐条勾选已完成项，Agent 在下次查询时只展示未完成项。

## E5：Prompt 模板管理

> 核心价值：让 Agent 在不同场景下说"不同的话"，提升回答的专业性和针对性。

---

### US-14：创建和激活 Prompt 模板

**Card**

> 作为**Prompt 模板管理员**，我想创建不同场景的 Prompt 模板（巡检模式/分析模式/应急模式）并一键激活，这样所有用户的 Agent 立刻切换到对应的回答风格，不用重新部署。

**Conversation**

Agent 在不同场景下应该有不同的"人格"：日常巡检要简洁，分析报告要详细，应急响应要结构化。管理员需要一种方式来管理这些"人格"——创建模板、一键切换、让所有用户立刻生效。

**Acceptance Criteria**

1. 管理员可通过后台界面创建 Prompt 模板，包含：模板名称、场景描述、Prompt 正文、适用场景标签。
2. 模板支持变量占位符（如 `{{region}}`、`{{time}}`、`{{weather}}`），激活时自动填充实际值。
3. 管理员可一键激活某个模板，激活后所有用户的 Agent 回答立刻使用该模板的风格。
4. 同一时间只能有一个模板处于激活状态，激活新模板自动停用旧模板。
5. 模板列表展示所有模板及其状态（激活/停用），支持搜索和按场景标签筛选。
6. 停用所有模板时，Agent 回退到默认系统 Prompt，不影响核心功能。

---

### US-15：Prompt 模板预览与测试

**Card**

> 作为**Prompt 模板管理员**，我想在激活模板前先预览效果——输入一个测试问题，看 Agent 用这个模板会怎么回答，这样我不怕上线后回答风格不对。

**Conversation**

管理员最怕的是"改了 Prompt 但不知道效果如何"。直接上线可能会影响所有用户的体验。需要一个沙盒预览功能，让管理员在激活前先测试模板效果。

**Acceptance Criteria**

1. 管理员在模板编辑页面可点击"预览"按钮，弹出测试对话框。
2. 在测试对话框中输入测试问题，Agent 使用当前编辑中的模板（而非已激活的模板）生成回答。
3. 预览回答附带模板变量替换后的完整 Prompt，方便管理员检查变量是否正确填充。
4. 管理员可以在预览中修改 Prompt 并重新测试，无需保存模板。
5. 预览功能不影响线上用户的 Agent 行为。

## E6：平台运维与 MCP 配置

> 核心价值：让运维人员能灵活管理 Agent 的数据源和工具，确保系统稳定运行。

---

### US-16：MCP Server 配置管理

**Card**

> 作为**平台运维人员**，我想在 `.env` 文件或管理界面中配置 MCP Server 的地址和参数，Agent 启动时自动连接，这样我不用改代码就能切换数据源。

**Conversation**

Agent 的能力取决于它能调用哪些 MCP Server（数据库查询、图表生成、网页搜索等）。运维人员需要一种简单的配置方式来管理这些数据源——添加新的、禁用有问题的、切换地址的。所有这些操作都不应该需要修改代码。

**Acceptance Criteria**

1. 运维人员可在 `.env` 文件中配置 MCP Server 列表（名称、地址、参数），Agent 启动时自动连接所有已配置的 Server。
2. 支持运行时动态添加新的 MCP Server，无需重启 Agent（热加载）。
3. 禁用某个 MCP Server 后，Agent 自动回退到内置能力，不影响核心功能（如禁用 `chart` Server 后，Agent 以纯文本表格形式返回数据）。
4. 配置变更记录在操作日志中（谁、何时、改了什么）。
5. Agent 启动时检测所有已配置 MCP Server 的连通性，不可用的标记为"离线"并在管理界面展示状态。

---

### US-17：Agent 工具调用监控

**Card**

> 作为**平台运维人员**，我想看 Agent 的工具调用日志——哪个 MCP Server 被调了几次、耗时多少、成功还是失败，这样我能发现性能瓶颈和故障点。

**Conversation**

Agent 在处理用户请求时会调用多个 MCP Server（查数据库、生成图表、搜索网页等）。运维人员需要知道这些调用是否正常、哪些慢、哪些经常失败——否则出了问题只能两眼一抹黑。

**Acceptance Criteria**

1. 管理界面展示 MCP Server 调用统计面板：每个 Server 的调用次数、平均耗时、成功率。
2. 支持按时间范围筛选（最近 1 小时 / 24 小时 / 7 天）。
3. 失败调用展示错误详情（错误类型、错误信息、触发时间）。
4. 平均耗时超过 5 秒的 MCP Server 标记为"慢查询"，高亮显示。
5. 支持导出调用日志为 CSV 文件，供离线分析。
