RAG 架构、数据流与原理说明

这个仓库本质上是一个项目生成器。它根据 CLI 选择生成 FastAPI + Next.js 项目， RAG 代码通过 Cookiecutter 模板按需写入生成结果。生成出的应用再负责文档上传、 解析、向量化、检索、引用和需求知识库业务流。

一页总览

CLI 配置 fastapi-fullstack create ... --rag --vector-store pgvector
ProjectConfig RAGFeatures 校验并转换生成选项
Cookiecutter 模板 根据 enable_rag / use_pgvector 渲染代码
生成应用 只包含本次选择的 RAG、DB、队列、前端代码
最关键的理解： fastapi_gen/ 是生成器自己的代码； template/{{cookiecutter.project_slug}}/ 是未来被生成出去的应用代码； {% if cookiecutter.use_pgvector %} 是生成时判断，不是 FastAPI 运行时判断。

仓库结构

生成器代码

fastapi_gen/ 负责解析 CLI 参数、交互式提问、校验配置、生成 Cookiecutter context，然后调用 Cookiecutter。

  • cli.py：命令入口
  • config.py：配置模型和校验规则
  • prompts.py：交互式问题
  • generator.py：实际渲染模板

应用模板

template/ 是要被复制和渲染出去的项目骨架，里面大量文件带 Jinja 条件。

  • backend/app/services/rag/：RAG 服务
  • backend/app/api/routes/v1/rag.py：通用 RAG API
  • backend/app/api/routes/v1/knowledge_bases.py：需求知识库 API
  • frontend/src/：Next.js 前端与 BFF routes

生成器层

生成器层决定“最终项目里会有什么代码”。它不直接执行 RAG 检索，而是把用户选择变成模板变量。

RAGFeatures 是生成选项，不是运行时服务

class RAGFeatures(BaseModel):
    enable_rag: bool = False
    enable_google_drive_ingestion: bool = False
    enable_s3_ingestion: bool = False
    reranker_type: RerankerType = RerankerType.NONE
    enable_image_description: bool = False
    pdf_parser: PdfParserType = PdfParserType.PYMUPDF
    vector_store: VectorStoreType = VectorStoreType.MILVUS

用户在 CLI 里选 --rag --vector-store pgvector 后， to_cookiecutter_context() 会展开成简单布尔值： 

{
  "enable_rag": True,
  "vector_store": "pgvector",
  "use_milvus": False,
  "use_qdrant": False,
  "use_chromadb": False,
  "use_pgvector": True
}

CLI 收集选择

fastapi-fullstack create demo --rag --vector-store pgvector 进入 fastapi_gen/cli.py。

ProjectConfig 校验组合

例如 Milvus/Qdrant 通常需要 Docker；pgvector 要求 PostgreSQL；Google Drive/S3 ingestion 必须先启用 RAG。

转换 Cookiecutter context

枚举值变成模板容易判断的布尔值，例如 use_pgvector=True。

渲染 template/

Jinja 条件裁剪掉不需要的代码，最终生成一个普通 FastAPI/Next.js 项目。

模板层

模板文件看起来像 Python/TypeScript，但里面有 Jinja 语句。 这些语句在生成项目时执行，生成后不会留给应用运行时判断。

模板中的代码

{%- if cookiecutter.enable_rag %}
from app.services.rag.retrieval import RetrievalService

{%- if cookiecutter.use_pgvector %}
from app.services.rag.vectorstore import PgVectorStore
{%- elif cookiecutter.use_milvus %}
from app.services.rag.vectorstore import MilvusVectorStore
{%- endif %}
{%- endif %}

选择 pgvector 后生成的代码

from app.services.rag.retrieval import RetrievalService
from app.services.rag.vectorstore import PgVectorStore

Milvus/Qdrant/Chroma 的 import 分支不会进入生成结果。生成后的应用更像“专用项目”，不是运行时大杂烩。

模板变量 含义 影响
enable_rag 是否生成 RAG 功能 控制 services/rag/、RAG API、RAG CLI、依赖项是否存在
use_pgvector 使用 PostgreSQL pgvector 生成 PgVectorStore 路径，不需要额外向量数据库容器
use_milvus 使用 Milvus 生成 Milvus client、Docker service 和对应配置
enable_reranker 是否启用 reranker 检索服务会注入 RerankService，查询时可二次排序
enable_google_drive_ingestion 是否生成 Google Drive connector 同步源 registry 中注册 gdrive

运行时服务层

生成后的 FastAPI 应用启动后，RAG 的核心服务通过 api/deps.py 装配。 路由层只处理 HTTP，真正逻辑放在 service/repository/worker。

DocumentProcessor

负责把 PDF、DOCX、TXT、MD 等文件解析成页面、Markdown-like 文本和 chunks。

EmbeddingService

统一 OpenAI、Voyage、Gemini、SentenceTransformers，输出固定维度向量。

BaseVectorStore

统一 Milvus、Qdrant、Chroma、pgvector 的插入、搜索、删除、列表接口。

IngestionService

编排解析、去重、embedding、写入向量库，并返回完整 Markdown。

RetrievalService

编排向量搜索、可选 BM25 hybrid、可选 rerank、分数过滤和去重。

RAGDocumentService

管理 SQL 中的文档状态、文件路径、完整 Markdown、版本和下载信息。

文档入库流

文档入库有两份存储目标：向量库保存可检索 chunks，SQL 保存文件状态和完整原文。 这是需求知识库后续能做“可信引用”的基础。

HTTP 上传 POST /api/v1/rag/collections/{name}/ingest 或 KB 上传接口
DB 追踪记录 RAGDocument.status=processing，保存文件元数据
后台任务 Celery / Taskiq / ARQ / in-process background
解析与 chunk DocumentProcessor.process_file()
Embedding EmbeddingService.embed_document()
写向量库 VectorStore.insert_document() 写 chunk、vector、metadata
保存 Markdown RAGDocument.markdown_content 保存完整解析结果
完成状态 status=done、vector_document_id、chunk_count
不要把向量库当成唯一真相。 向量库里的 chunk 是为了召回；需求知识库的事实依据来自 SQL 中保存的完整 Markdown 原文。

检索问答流

通用 RAG 搜索和需求知识库问答不是同一层。通用 RAG 返回相似 chunk； Req KB 会进一步映射回完整 Markdown，组织带来源的业务回答。

通用 RAG 搜索

  1. 用户调用 POST /api/v1/rag/search
  2. RetrievalService.retrieve() 接收 query
  3. 调用 VectorStore.search() 做相似度搜索
  4. 可选 BM25 hybrid 和 reranker
  5. 按分数过滤、去重、返回 chunks

需求知识库查询

  1. 用户调用 POST /api/v1/kb/{kb_id}/query
  2. 先用 query variants 做向量检索
  3. 用 vector_document_id 找最新 RAGDocument
  4. 在完整 markdown_content 中定位章节证据
  5. 没有向量命中时走 Markdown fallback
  6. 生成 [来源: 文件 > 章节 > page/chunk] 引用
用户问题
  -> query variants
  -> vector search
  -> hit.parent_doc_id
  -> RAGDocument.vector_document_id
  -> RAGDocument.markdown_content
  -> 章节证据 + 引用
  -> RequirementAIService 可选组织中文回答

需求知识库层

Req KB 是建立在通用 RAG 之上的业务层。它把“文档检索”变成“需求录入、查询、拆解、变更、版本管理”。

能力 接口 核心服务 原理
一句话创建需求 POST /api/v1/kb/{kb_id}/requirements/from-text RequirementWorkflowService.create_from_text AI 或 fallback 生成 Markdown，直接写入 RAGDocument
原文引用查询 POST /api/v1/kb/{kb_id}/query RequirementQueryService.query_kb 向量定位 + Markdown 原文校验 + 来源标签
需求拆解 GET /api/v1/kb/{kb_id}/documents/{doc_id}/breakdown RequirementWorkflowService.break_down_document 按 Markdown heading 切章节，并输出来源
修改建议/草稿/应用 POST /api/v1/kb/{kb_id}/documents/{doc_id}/change RequirementWorkflowService.change_document 产品可生成新版，开发/测试偏建议或草稿
版本与 diff versions / diff / apply-draft RequirementWorkflowService previous_version_id 串版本链，is_latest 标当前版
Req KB 的核心设计： RAG 负责召回可能相关的文档位置；业务层必须回到完整 Markdown 原文后再回答。 这样可以减少只凭 chunk 片段误答的问题，也方便做版本、diff 和审批。

同步源

同步源是“批量把外部文档送进同一条 ingestion pipeline”的入口。 Google Drive 和 S3 connector 只负责列文件、下载文件、提供稳定的 source_path 去重键。

SyncSource 保存 connector 类型、collection、config、schedule
Connector Registry gdrive / s3 根据生成选项注册
下载临时文件 download_file() 到临时目录
复用入库链路 IngestionService.ingest_file()

所以新增一个同步源时，通常不需要重写 RAG。只要实现 BaseSyncConnector.list_files() 和 download_file()， 再注册到 CONNECTOR_REGISTRY。

核心数据模型

向量库中的数据

  • id：chunk id
  • parent_doc_id：解析出的 document id
  • content：chunk 文本
  • embedding / vector：向量
  • metadata：filename、page_num、chunk_num、source_path、content_hash

SQL RAGDocument

  • id：业务追踪记录 id
  • collection_name：向量 collection
  • status：processing / done / error
  • vector_document_id：向量库 parent_doc_id
  • markdown_content：完整 Markdown 原文
  • version、is_latest、previous_version_id：需求版本链
  • knowledge_base_id、organization_id：团队/KB 归属
两个 id 要区分： RAGDocument.id 是 SQL 业务记录； vector_document_id 对应向量库中的 parent_doc_id。 查询从向量 hit 回 SQL 时靠这个关联。

当前边界

已经形成闭环

  • 生成器可按配置生成 RAG 项目
  • 支持多种向量库和 embedding provider
  • 文件上传、后台入库、搜索、同步源已成型
  • Req KB MVP 支持中文需求录入、查询、拆解、变更和版本
  • 查询回答能回到完整 Markdown 原文生成来源引用

还不是完整生产态

  • 多轮澄清状态机仍是后续工作
  • 结构化红绿 diff 审批 UI 还未完整
  • Redis/pubsub 跨进程通知 fan-out 仍需补齐
  • read receipts / 通知中心仍未完整
  • 旧版本向量清理和生产登录角色映射还需要硬化

最短心智模型

生成器阶段:
  CLI flags -> ProjectConfig/RAGFeatures -> cookiecutter context -> template 渲染 -> 生成项目

运行时入库:
  file -> RAGDocument(status=processing) -> worker -> parse/chunk -> embedding -> vector store
       -> RAGDocument(status=done, markdown_content, vector_document_id)

运行时查询:
  query -> vector hits -> 找 RAGDocument -> 读 markdown_content -> 生成来源引用 -> 可选 AI 组织回答

需求知识库:
  通用 RAG + 完整 Markdown + 角色权限 + 版本链 + 变更审批
